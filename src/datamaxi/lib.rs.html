1//! # DataMaxi+ Rust SDK
2//!
3//! This is the official implementation of Rust SDK for [DataMaxi+](https://datamaxiplus.com/).
4//! The package can be used to fetch both historical and latest data using [DataMaxi+ API](https://docs.datamaxiplus.com/).
5//!
6//! - [Installation](#installation)
7//! - [Configuration](#configuration)
8//! - [Links](#links)
9//! - [Contributing](#contributing)
10//! - [License](#license)
11//!
12//! ## Installation
13//!
14//! ```shell
15//! [dependencies]
16//! datamaxi = { git = "https://github.com/bisonai/datamaxi-rust.git" }
17//! ```
18//!
19//! ### Minimum Supported Rust Version (MSRV)
20//!
21//! This crate requires **Rust 1.86** or newer. The MSRV is verified in CI and
22//! may be raised in a minor version bump.
23//!
24//! ## Configuration
25//!
26//! Private API endpoints are protected by an API key.
27//! You can get the API key upon registering at <https://datamaxiplus.com/auth>.
28//!
29//!
30//!| Option     | Explanation                                                                   |
31//!|------------|-------------------------------------------------------------------------------|
32//!| `api_key`  | Your API key                                                                  |
33//!| `base_url` | If `base_url` is not provided, it defaults to `https://api.datamaxiplus.com`. |
34//!
35//! ## Examples
36//!
37//! ### CEX Candle
38//!
39//! The client is async by default (requires a runtime such as `tokio`). For a
40//! synchronous client, enable the `blocking` feature and use the mirrored
41//! wrappers under `datamaxi::blocking` with `datamaxi::api::blocking`.
42//!
43//! ```no_run
44//! use datamaxi::{
45//!     CexCandleExchangesMarket, CexCandleMarket, CexCandleOptions, CexCandleSymbolsOptions,
46//!     Client,
47//! };
48//!
49//! # async fn run() -> Result<(), Box<dyn std::error::Error>> {
50//! let client = Client::new("my_api_key");
51//! let candle = client.cex_candle();
52//!
53//! // Supported exchanges, symbols and intervals
54//! let _ = candle.exchanges(CexCandleExchangesMarket::Spot).await?;
55//! let _ = candle.symbols("binance", CexCandleSymbolsOptions::new()).await?;
56//! let _ = candle.intervals().await?;
57//!
58//! // Fetch CEX candle data
59//! let _ = candle
60//!     .get(
61//!         "binance",
62//!         "BTC-USDT",
63//!         CexCandleOptions::new().market(CexCandleMarket::Spot),
64//!     )
65//!     .await?;
66//! # Ok(())
67//! # }
68//! ```

96/// Typed wrappers for every REST endpoint on the data API — the canonical
97/// surface (CEX candle, OI, Liquidation, cex-symbol, …). Endpoint groups are
98/// reached through accessors on the root [`Client`]. Async by default; with the
99/// `blocking` feature, a parallel [`blocking`] module offers synchronous
100/// equivalents.
101///
102/// ```no_run
103/// use datamaxi::{Client, LiquidationHeatmapOptions};
104///
105/// # async fn run() -> Result<(), Box<dyn std::error::Error>> {
106/// let client = Client::new("YOUR_API_KEY");
107/// let heatmap = client
108///     .liquidation()
109///     .heatmap(LiquidationHeatmapOptions::new())
110///     .await?;
111/// # Ok(())
112/// # }
113/// ```
114pub use generated::*;
115
116/// The root client and its builder are re-exported at the crate root so callers
117/// write `datamaxi::Client` / `datamaxi::ClientBuilder`. Endpoint groups hang
118/// off the client via generated accessors, e.g. `client.cex_candle()`.
119pub use api::{Client, ClientBuilder};
